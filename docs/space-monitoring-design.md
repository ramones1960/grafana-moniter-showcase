# 宇宙開発分野における Grafana 監視設計

本デモが想定する「宇宙ミッション運用センター（Mission Operations Center）」の監視設計をまとめる。

## 全体アーキテクチャ

```
   ┌──────────────────────────┐
   │   Telemetry Simulator     │  模擬衛星 + 軌道 + 地上局
   │  (orbit / power / thermal │
   │   / attitude / comms)     │
   └───┬───────────┬───────┬───┘
       │ 高頻度TM   │ イベント│ /metrics
       ▼           ▼       ▼
  ┌─────────┐ ┌──────────┐ ┌──────────┐
  │InfluxDB │ │TimescaleDB│ │Prometheus│
  │(1Hz TM) │ │(パス/EO)  │ │(監視)    │
  └────┬────┘ └─────┬────┘ └────┬─────┘
       └───────┬────┴───────────┘
               ▼
          ┌─────────┐
          │ Grafana │  4 ダッシュボード
          └─────────┘
```

## なぜデータソースを 3 つに分けるのか

宇宙機テレメトリは性質の異なる 3 種類のデータが混在する。**それぞれに最適な DB**を割り当てる。

1. **高頻度連続テレメトリ**（電源電圧・温度・姿勢レート, 1Hz〜）
   → **InfluxDB**。高い書き込みスループットとダウンサンプリング・長期保存に強い。Flux で柔軟に集計。

2. **離散イベント・運用台帳**（可視パスの AOS/LOS、コンタクト実績、地球観測キャプチャ、異常記録）
   → **TimescaleDB**。リレーショナルな結合（地上局マスタ × パス実績）と時系列を両立。SQL で運用レポート。

3. **システム・業務メトリクス**（衛星ヘルスのサマリ、シミュレータ自身のリソース）
   → **Prometheus**。アラートルールと相性が良く、FDIR 的な閾値監視に最適。

## 模擬する宇宙機モデル

太陽同期軌道（SSO）の地球観測衛星 1 機を想定。

| サブシステム | モデル化する量 |
|---|---|
| 軌道 (Orbit) | 円軌道伝播 → 直下点(lat/lon)・高度・食(eclipse)・β角 |
| 電源 (EPS) | 太陽電池発電（日照依存）・バッテリー SoC 積分・バス電圧 |
| 熱 (TCS) | ゾーン別温度（日照/食で変動: バッテリ/OBC/ペイロード/パネル） |
| 姿勢 (ADCS) | 角速度・ポインティング誤差・リアクションホイール回転数 |
| 通信 (COMM) | 地上局可視時のダウンリンク・SNR（仰角依存）・信号ロック |
| ミッション | 地上局パス生成・地球観測ターゲット撮像・データバッファ |

## 地上局ネットワーク

代表的な極域・中緯度局を模擬（実在局を参考にした座標）:

- 〜 Svalbard (Norway, 78°N) — 極軌道で全周回可視に近い
- 〜 Tokyo / Katsuura (Japan)
- 〜 Fairbanks (Alaska)
- 〜 Santiago (Chile)

衛星がいずれかの局から仰角 > 最低仰角（既定 5°）で見えるときに「パス（コンタクト）」が成立し、
TimescaleDB に AOS/LOS と最大仰角・ダウンリンク量が記録される。

## アラート設計（FDIR 相当）

`stack/prometheus/alerts.yml` に実装:

| アラート | 条件 | 重大度 |
|---|---|---|
| BatteryLow | SoC < 30% | warning |
| BatteryCritical | SoC < 20% | critical |
| ThermalHigh | 任意ゾーン > 上限温度 | warning |
| AttitudeLoss | ポインティング誤差 > 5° | critical |
| DataBufferFull | オンボード蓄積 > 90% | warning |
